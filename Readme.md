# Этапы развития проекта
 - *Bootloader* : **расширить загрузчик: загрузка ядра с диска, переключение в protected mode**
 - *Kernel* : **Базовое ядро на Си**
 - *VGA driver* : **Вывод текста на экран**
 - *GDT* : **Глобальная таблица дескрипторов**
 - *PMM* : **Физический менеджер памяти**
 - *VMM* : **Виртуальный менеджер памяти / Paging**
 - *kheap* : **Аллокатор кучи ядра**

 - *IRQ* : **Обработка прерываний, таймер**

## Как это запускать?
Для начала вам надо скомпилировать кросс-компилятор, приведу пример для убунту, на остальные дистры будет всё точно также. Вам понадобится собрать i686-elf-gcc из исходников

        sudo apt install build-essential bison flex libgmp3-dev libmpc-dev libmpfr-dev texinfo wget
        export PREFIX="$HOME/opt/cross"
        export TARGET=i686-elf
        export PATH="$PREFIX/bin:$PATH"
        mkdir -p $PREFIX

        cd ~/tmp
        wget https://ftp.gnu.org/gnu/binutils/binutils-2.41.tar.gz
        tar xf binutils-2.41.tar.gz
        mkdir build-binutils && cd build-binutils

        ../binutils-2.41/configure \
            --target=$TARGET \
            --prefix=$PREFIX \
            --with-sysroot \
            --disable-nls \
            --disable-werror

        make -j$(nproc)
        make install

        cd ~/tmp
        wget https://ftp.gnu.org/gnu/gcc/gcc-13.2.0/gcc-13.2.0.tar.gz
        tar xf gcc-13.2.0.tar.gz
        mkdir build-gcc && cd build-gcc
        ../gcc-13.2.0/configure \
            --target=$TARGET \
            --prefix=$PREFIX \
            --disable-nls \
            --enable-languages=c \
            --without-headers
        make -j$(nproc) all-gcc
        make -j$(nproc) all-target-libgcc
        make install-gcc
        make install-target-libgcc

После того как кросс-компилятор собран, просто:

        make        # собрать os.img
        make run    # запустить в QEMU
        make clean  # почистить артефакты сборки

- **Что такое А20?**

Если говорить просто: это переключатель, который позволяет процессору видеть больше 1 МБ оперативной памяти. Почему так? Всё просто, наш любимый Х86. В первых процессорах Intel 8086 было всего 20 адресных линий (от A0 до A19). Это позволяло адресовать ровно 2^20 байт = 1 МБ.
Позже начали появляться новые процессоры и надо было оставить обратную совместимость, старые программы полагались на wrap-around памяти. Если они обращались к адресу чуть выше 1 МБ, процессор 8086 аппаратно отбрасывал лишний бит и возвращал их в начало первого мегабайта, то есть было фундаментальное ограничение размера ОЗУ процессором. Процессор 80286 этого не делал и лез по адресу выше 1 МБ. Это ломало старый софт.
Решением стал логический вентиль на 21-ю линию (A20), который по умолчанию был выключен. Чтобы твое ядро могло использовать всю оперативку, а не только первый мегабайт, эту линию нужно активировать вручную как в boot.asm. В real mode адрес вычисляется как segment * 16 + offset, это означает что максимальный адрес может быть 0xFFFF:0xFFFF. В десятичном виде это 65535 * 16 + 65535 = 1,114,095 байт (чуть больше 1 МБ). Т.е. если A20 выключена то 21-й бит адреса аппаратно игнорируется. Адрес 1,000,500 превращается в 000,500. Процессор wrap-around'ится в начало первого мегабайта. Но если A20 включена то процессор видит разницу между 000,500 и 1,000,500. В моем проекте юзается способ через клавиатурный контроллер: проверяем через сравнение двух адресов. Если разные — A20 уже включен. Если нет — включаем через клавиатурный контроллер (порт 0x64/0x60).

- **GDT (Global Descriptor Table)**

Это таблица которая объясняет процессору как сегментирована память. Если в 16-битном пространстве Real mode мы придерживались формуле segment * 16 + offset, то в 32-битном пространстве Protected mode сегментные регистры cs, ds, ss уже не содержат адреса. Они содержат селектор — то есть индекс в таблице GDT. Можно заметить что Base (базовый адрес) и Limit (размер) разбросаны по разным байтам дескриптора:
1) Base разбит на: 2 байта (dw), 1 байт (db) и еще 1 байт (db).
2) Limit разбит на: 2 байта (dw) и 4 бита внутри байта Granularity.

Когда Intel переходила с 16-битного 80286 на 32-битный 80386, им нужно было сохранить обратную совместимость. Они не могли просто взять и расширить дескриптор, поэтому костыльно впихивали новые биты адреса в пустые места старого формата. В итоге получился винегрет который мы видим в коде. Может есть более красивое решение, но я честно не нашёл и не врубился всецелом. И так по порядку:

Нулевый дескриптор `dq 0x0000000000000000` — процессор требует, чтобы первый дескриптор всегда был заполнен нулями. Если случайно забыть инициализировать сегментный регистр (он будет равен 0) и попытаешься обратиться к памяти, процессор увидит что ты пытаешься использовать нулевый селектор и выбросит исключение. Это защита от багов. У меня используется Flat Model: Base = 0x00000000, Limit = 0xFFFFFFFF — сегменты кода и данных накладываются друг на друга и покрывают всю доступную память. По этому же принципу работают виндовс и линух, т.к. реальное разграничение памяти сейчас делается через Paging, а не через сегментацию.

*   gdt_descriptor — процессору нужно знать где лежит таблица. Для этого есть специальная инструкция lgdt. Она принимает на вход адрес структуры с размером таблицы и её физическим адресом.*

*   Селекторы CODE_SEG и DATA_SEG — это смещения внутри GDT. Каждый дескриптор занимает 8 байт. 0x00 — Null; 0x08 — Code; 0x10 — Data.*

- **Зачем нужен PMM?**

Когда компьютер включается, в нём есть оперативная память — допустим, 128 мегабайт. Это просто огромный массив байтов. Никто не следит за тем, какие части этого массива уже используются, а какие свободны. Ядро операционной системы должно само организовать учёт: вот этот кусок памяти занят под код ядра, вот этот — под данные, вот этот — свободен, можно отдать кому-нибудь. PMM — Physical Memory Manager — тот кто этим занимается. Его задача предельно проста: знать какие страницы памяти свободны, а какие заняты. Выдавать страницу по запросу и принимать обратно при освобождении.

Почему страницы, а не просто байты? Потому что управлять памятью побайтово дорого. Процессор работает с памятью блоками по 4KB — это называется страница. Как хранить информацию о том, какие страницы свободны? Самый простой способ — bitmap, где каждый бит отвечает за одну страницу. 32768 страниц = 32768 бит = 4096 байт = всего 4KB. В восемь раз экономнее чем по байту на страницу.

*   В этом проекте bitmap живёт по адресу 0x1000. Первые 4 страницы (16 KB) резервируются под код ядра и сам bitmap при инициализации.*

- **VMM (Virtual Memory Manager) и Paging**

После PMM, который работает с физическими страницами, нужен ещё один слой — VMM, который занимается виртуальными адресами. Смысл вот в чём: каждый процесс думает что у него есть собственные 4 ГБ памяти начиная с нуля. На самом деле это иллюзия — процессор прозрачно транслирует виртуальные адреса в физические через специальные таблицы.

Эти таблицы называются Page Directory и Page Table. Архитектура двухуровневая: Page Directory содержит 1024 записи, каждая из которых указывает на Page Table, а та в свою очередь тоже содержит 1024 записи — и каждая уже указывает на реальную физическую страницу 4KB. Итого 1024 × 1024 × 4KB = 4 ГБ адресного пространства.

Виртуальный адрес разбирается так:
- Биты 31:22 - индекс в Page Directory (10 бит)
- Биты 21:12 - индекс в Page Table (10 бит)
- Биты 11:0 - смещение внутри страницы (12 бит)

Чтобы включить paging, нужно: создать Page Directory, замаппировать нужные страницы, записать адрес PD в регистр CR3, и выставить бит PG (31-й) в регистре CR0. После этой инструкции все адреса начинают транслироваться, включая тот код который эту инструкцию только что выполнил. Поэтому перед включением paging'а нужно обязательно замаппировать код ядра identity (virtual == physical), иначе следующая же инструкция вызовет page fault и всё упадёт.

*   VGA буфер по адресу 0xB8000 тоже нужно замаппировать отдельно, иначе экран перестанет работать после включения paging'а.*

*   TLB (Translation Lookaside Buffer) — это кеш трансляций в процессоре. При анмаппировании страницы нужно сбросить её запись в TLB через инструкцию invlpg, иначе процессор может использовать устаревшую трансляцию.*

- **kheap**

PMM выдаёт целые page'ы по 4KB. Это удобно для крупных аллокаций, но неудобно если нужно выделить 100 байт, тратить ради этого весь page тупо. Для этого поверх PMM сделан kheap. Это хип ядра с kmalloc/kfree.

Устроено это через linked list блоков. Каждый блок в памяти начинается с заголовка `block_header_t` где хранится размер блока, флаг занятости и указатели на соседние блоки. При kmalloc ищем первый подходящий свободный блок, если нашелся таков то отдаём. Если нет расширяем кучу с помощтю новых pageв. При kfree помечаем блок свободным и склеиваем его с соседними свободными блоками. Куча растёт от 0x400000 до 0x4000000 (64 MB --- пока что).

*   Все аллокации выравниваются на 16 байт, это требование многих алгоритмов и SIMD инструкций*

*   Coalesce - это по сути объединение соседних свободных блоков. Он уже есть в kheap, чтобы долгоживущие аллокации меньше фрагментировали память.*

- **VGA driver**

Текстовый режим VGA это по своей сути буфер по адресу 0xB8000. 80 × 25 = 2000 ячеек по 2 байта каждая. Младший байт аски символ, старший - атрибут цвета. Цвет упаковывается в один байт:: биты 3:0 - foreground 16 цветов, биты 7:4 — background 16 цветов. Все в лоб, если хочешь вывести символ то записывай в нужную ячейку. Никаких сисколов, никакого посредника

Курсор управляется через порты 0x3D4/0x3D5, индексный и дата регистры CRT контроллера соответственно. Позиция курсора равна y × 80 + x, передаётся как два байта через регистры 0x0E (верхний) и 0x0F (нижний)

*   volatile uint16_t* на буфере очень важен, ибо без него компилятор может решить закешировать запись и вообще не отправить её в железо. volatile гарантирует что каждая запись действительно уйдёт по адресу, этой конструкции будет довольно много в проекте*

- **Keyboard**

Клавиатура работает через PS/2 контроллер. На порт 0x60 приходят данные, читать scancode, порт 0x64 это статус. Бит 0 в статусном регистре означает что в буфере есть данные для чтения. Важно омтетить что Scancode это не ASCII, это номер физической клавиши. Таблица трансляции scancode -> ASCII хранится прямо в коде (US QWERTY раскладка)
Key release события отличаются от key press тем что бит 7 scancode установлен. То есть нажатие 'A' даёт 0x1E, а отпускание — 0x9E.

Внутри драйвера сделан circular буфер на 256 элементов. keyboard_handler пишет в него из прерывания, а читается он из основного кода. Пока IDT не настроена значит keyboard_handler вызвать невозможно. Но keyboard_getchar через hlt-loop будет работать как только прерывания появятся

*   Через тот же контроллер клавиатуры, порт 0x64, в бутлоадере включается A20/// вот такой занятный факт.*

- **Timer и PIC**

PIC - Programmable Interrupt Controller : это микросхема которая управляет прерываниями. В IBM PC их две штуки: мастер (oldschool) PIC1 и слейв PIC2, касдкадные через линию IRQ2. Всего они обрабатывают 16 линий прерываний IRQ 0–15

Проблема следующая: по умолчанию IRQ 0–7 мапятся на прерывания 0x08–0x0F, а это как раз зона эксептов процессора (деление на ноль, page fault и т.д.). Коллизия. Поэтому одно из первых что нужно сделать - перемапить PIC. Делается это через последовательность команд инициализации ICW1-ICW4 в порты 0x20/0x21-мастер и 0xA0/0xA1-слейв. После ремапинга IRQ 0–7 становятся прерываниями 0x20–0x27, а IRQ 8–15 0x28–0x2F. В моем проекте PIC ремаппится в timer.c.

PIT (Programmable Interval Timer) - это таймер который тикает с фиксированной частотой. Базовая частота 1.193182 МГц, делится на делитель который мы сами задаём. Хотим 100 тиков в секунду значит пишем делитель 11932 в порт 0x40. По каждому тику прилетает IRQ0 -> прерывание 0x20 -> вызывается timer_handler.

*   После каждого обработанного прерывания нужно отправить End Of Interrupt - EOI командой 0x20 в порт PIC, иначе контроллер думает что прерывание ещё обрабатывается и блочит все последующие*

- **Disk (ATA PIO)**

Жёсткий диск подключается по интерфейсу ATA (он же IDE). Режим PIO (Programmed I/O) самый простой. CPU сам читает/пишет данные через порты ввода-вывода 0x1F0–0x1F7. Медленно и не спеша, с некой иронией, но зато понятно и не требует DMA.

Адресация через LBA28 Logical Block Addressing: диск воспринимается как плоский массив секторов по 512 байт. LBA28 это 28 битный номер сектора, что даёт адресацию до 128 ГБ (2^28 × 512 байт).

Последовательность чтения: выбрать диск -> записать адрес сектора по кускам в регистры 0x1F3 – 0x1F6 -> послать команду READ (0x20) в 0x1F7 -> ждать бит DRQ в статус регистре -> читать 256 слов через inw(0x1F0) = 512 байт.

Команда IDENTIFY (0xEC) позволяет узнать всё о диске: модель, серийник, размер. Контроллер возвращает 256 слов с информацией. Что интересно, так это то что строки там хранятся побайтово переставленными, это legacy с древних времён ATA

*   DRQ (Data Request) — это бит в статусном регистре означающий что контроллер готов отдать данные. Читать раньше чем он установится — бессмысленно.*

*   Команда `disk` в терминале вызывает ATA IDENTIFY и показывает найденные primary master/slave устройства.*

- **IDT (Interrupt Descriptor Table)**

IDT - это аналог GDT но для прерываний. GDT объясняет процессору как сегментирована память, IDT объясняет что делать когда происходит прерывание или исключение. Таблица на 256 записей состоит из 32 исключения процессора (деление на ноль, page fault, general protection fault и тд) + IRQ от устройств + программные прерывания

Каждая запись в IDT это дескриптор прерывания: адрес обработчика, сегмент кода, флаги (тип шлюза, уровень привилегий). Когда происходит прерывание, процессор смотрит в IDT, берёт адрес обработчика и прыгает туда, предварительно сохранив состояние (pushad + сегментные регистры) на стеке.

В проекте уже есть ISR/IRQ stubs в ассемблере, заполнение IDT, `lidt`, ремап PIC и включение `sti`. Благодаря этому таймер и клавиатура работают через настоящие IRQ, а терминал может жить в обычном цикле с `hlt`, просыпаясь по прерываниям.

*   Как процессор работает с памятью? (из книги Макарова и Насырова)*
*   В процессорах начиная с i386 базовые регистры стали 32-битными, что позволяет адресовать до 4 гигабайт. Сегментные регистры остались 16-битными, но в защищённом режиме они не содержат адреса — они содержат индекс дескриптора. Таблица дескрипторов представляет собой массив записей по 8 байт, где описываются адреса, пределы и права доступа к сегментам. При обращении за пределы сегмента процессор генерирует исключение. При обращении к сегменту из дескриптора берётся базовый адрес и складывается со смещением — получается линейный 32-битный адрес. Для i386 и выше этот линейный адрес проходит ещё через трансляцию paging'а и становится физическим.*
